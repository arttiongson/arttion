I ran $65 million in store operations before I ever wrote a line of code. That's why I build differently, and why I now direct AI agents to ship the systems I used to wish for.

Before moving into product accounting, I led store operations at a major national retailer, managing a $65M book of business, leading cross-functional teams, redesigning operational workflows, training staff on new systems, and owning the P&L outcomes.

I know what operational friction looks like because I've spent years fighting it from the inside.

When I moved into product accounting, I found myself working inside Microsoft Dynamics NAV, running monthly close processes and managing inventory reconciliation. Within my first month, I looked at a reconciliation workflow that consumed 2-3 days every cycle and I decided to fix it.

I fixed it. I directed AI to build an automated Power Query pipeline pulling live data through ODBC connections directly from our ERP database. What took days now takes one click. The real lesson was not that AI could write code for me. It was that I could direct it to build real systems, and that the constraint was my judgment about what to build, not the typing.

How I Work

I don't write the code, and I never have. I direct AI agents to build it. My job is the part the agents can't do: owning the problem, setting the architecture, holding the standards, and verifying the result against the source.

01 Own
I own the problem end to end
Requirements, architecture, data decisions, and the definition of done are mine. I come from operations and accounting, so I read process bottlenecks fast and know what a correct result actually looks like before a single line is written.
02 Direct
I direct agents, in parallel
I orchestrate multiple AI coding agents at once, each in its own lane, coordinated through written handoff docs and strict version control. When two agents once built the same component independently, I caught the collision in git, backed out the duplicate, and kept the better implementation.
03 Govern
I set the standards they work within
The agents operate inside rules I define: deny-by-default permissions, propose-don't-apply self-modification, and test-before-promote. I also build the custom tooling (MCP servers) that extends what they can safely do.
04 Verify
I verify against the source
Agents are fast, not infallible. I validate every output against the real data. That discipline is how I caught an accounting process overstating a material line by ~57% and surfaced redundant spend others had missed.

Self-Hosted AI Platform · Always on · Private project

Jarvis

A private, always-on AI assistant running on my own hardware: full-duplex voice, two-way iMessage, smart-home control, persistent memory, and the ability to safely propose changes to its own code. Not a wrapper around a chatbot, but a distributed, self-improving platform with its own security model, observability stack, automated test gate, and a daily self-reflection loop. Claude is the reasoning layer. I architected and directed it end to end, specifying the systems and standards while AI coding agents wrote the implementation.

Architecture & key design decisions

Topology
Distributed, augment-don't-replace
Multiple machines on a LAN, each with a deliberate role: a canonical production host running the home-automation hub, voice orchestration, and state; an ML coprocessor that bridges the Apple ecosystem; and a separate, fail-closed test environment. I kept the orchestrator on Linux and treated the Apple machine as a capability endpoint rather than consolidating, since sub-millisecond LAN round-trips make the split nearly free while keeping the right OS for each job.
Security
Governed agent runtime
The brain is an LLM with tool access, sandboxed by a deny-by-default permission gate: every tool call is explicitly allow-listed and audited, with no ambient access and per-call authorization. The model was hardened iteratively, closing over-permissive paths as I found them, around one core invariant: the assistant cannot widen its own sandbox.
Cost & quality
Two-tier model routing
A semantic capability and stakes router picks a cheap, fast model for simple turns and escalates to the stronger model for hard or agentic ones, per turn and fail-open. It holds a heavy-use assistant inside a fixed monthly budget (roughly a 3x cost difference between tiers) without dumbing down hard requests.
Self-improvement
Self-modification, human-gated
The assistant can propose edits to its own code, persona, and docs. Each proposal is automatically fitness-tested (compiles, imports, and does not alter its own guardrails) and archived in version control, but nothing applies without my approval. Anchor-based patches let it edit one section of a large file instead of regenerating it.
Memory
Three-tier memory & curation
Hot (how it responds), cold (what it knows), and identity layers, all git-versioned and edited through gated tools. A daily reflection job distills lessons, consolidates the knowledge base, and evicts stale entries, autonomous but reversible and LLM-judged.
Voice
Latency-tuned voice pipeline
Hardware acoustic-echo-cancellation enables full-duplex and barge-in, with streaming speech-to-text and text-to-speech. I drove first-audio latency to about 0.2 seconds through per-stage optimization: eliminating dead narration paths, cache-warming, low-latency audio buffering, and TTS engine bake-offs.
Observability
Audit stack & a real test gate
A six-table audit database, health and metrics endpoints, and custom latency and audit tooling. A fail-closed automated test harness proves the test environment never touches production state, runs as the pre-deploy gate, and includes both deterministic checks and LLM-judged behavioral scenarios.
Self-map
The system documents itself
An indexed map of every subsystem, plus skills the assistant uses to navigate and reason about its own architecture. A daily audit flags documentation that has drifted from reality, so the map stays trustworthy as the system evolves.

Open Source · Model Context Protocol Server

Excel MCP Server

An open-source Model Context Protocol server that gives an AI agent direct read and write access to Excel workbooks: open a file, inspect sheets, read ranges, and write cells back, all as structured MCP tool calls. The hard problem with spreadsheets is size, so the server adds token guardrails that keep large-file operations inside the model's context window instead of failing them. Built in Python.

What it does

An AI agent that can reason about a spreadsheet is only useful if it can actually touch one. This server exposes Excel as a set of structured tools an agent can call: list sheets, read a range, write cells, and inspect structure, so the agent works directly against a real workbook rather than a copy pasted into a prompt. It runs locally over the Model Context Protocol, the same interface any MCP-aware client (such as Claude) can connect to.

Key design decisions

The core problem
Spreadsheets overrun context
A single large workbook can blow past a model's context window in one naive read, which fails the whole operation. The interesting engineering is not the file I/O, it is making large-file access survivable for a model with a finite context budget.
The fix
Token guardrails that bound & page
Each operation bounds and pages what it returns, so a read of a huge sheet comes back in context-sized chunks instead of one oversized payload. Large-file work stays inside context limits rather than failing outright.
Interface
Structured MCP tool calls
Every capability is exposed as a typed MCP tool, so any MCP-aware agent can discover and call it with structured arguments and structured results, no bespoke glue per client.
Why I built it
From my own workflow
My day-to-day is product costing and inventory in spreadsheets. I wanted agents to operate on real workbooks safely, so I built the tool I needed and open-sourced it.
